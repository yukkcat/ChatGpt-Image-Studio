# 10万并发出图 - 架构分析与二改方案

## 目标

- 10万并发图片生成请求
- Free 账号出图
- 出图互不影响，前端交互丝滑
- 高稳定性，失败自动重试

---

## 一、出图完整流程

```
用户请求 POST /v1/images/generations
        │
        ▼
┌─────────────────────────┐
│  1. 准入控制 (Admission) │  ← 全局并发门控，超过上限排队/拒绝
│  channel-based gate     │
└──────────┬──────────────┘
           │
           ▼
┌─────────────────────────┐
│  2. 账号选择 (Acquire)   │  ← 从号池中轮询选一个可用 free 账号
│  lease 机制锁定账号      │     每账号同时只允许 N 个并发请求
└──────────┬──────────────┘
           │
           ▼
┌─────────────────────────┐
│  3. Bootstrap           │  ← 访问 chatgpt.com 首页，提取 PoW 脚本
│  获取浏览器指纹资源      │
└──────────┬──────────────┘
           │
           ▼
┌─────────────────────────┐
│  4. Chat Requirements   │  ← POST /backend-api/sentinel/chat-requirements
│  获取 sentinel token    │     解 PoW 挑战 + turnstile
│  + 解 PoW 证明          │
└──────────┬──────────────┘
           │
           ▼
┌─────────────────────────┐
│  5. Prepare Conversation│  ← POST /backend-api/f/conversation/prepare
│  获取 conduit_token     │     告诉上游"我要生图"
└──────────┬──────────────┘
           │
           ▼
┌─────────────────────────┐
│  6. Start SSE Stream    │  ← POST /backend-api/f/conversation
│  发起图片生成请求        │     流式接收事件，等待图片完成
│  (耗时 30-120 秒)       │
└──────────┬──────────────┘
           │
           ▼
┌─────────────────────────┐
│  7. Poll / Extract      │  ← 从 SSE 流中提取 file_id
│  获取图片 file_id       │     或 poll GET /conversation/{id}
└──────────┬──────────────┘
           │
           ▼
┌─────────────────────────┐
│  8. Download URL        │  ← GET /backend-api/files/{id}/download
│  解析图片下载地址        │     获取临时 Azure Blob URL
└──────────┬──────────────┘
           │
           ▼
┌─────────────────────────┐
│  9. 缓存 & 返回         │  ← 下载图片到本地缓存
│  返回给用户              │     返回本地 URL 或 base64
└─────────────────────────┘
```

### 关键耗时分布（单次请求）

| 阶段 | 耗时 | 说明 |
|------|------|------|
| Bootstrap | 1-3s | 首页预热 |
| Chat Requirements + PoW | 1-2s | sentinel token |
| Prepare | 0.5-1s | conduit token |
| SSE + 生图 | 30-120s | **主要瓶颈**，上游生图时间 |
| Poll (如需) | 3-30s | 异步模式下轮询 |
| Download URL | 0.5-1s | 获取下载地址 |
| **总计** | **35-160s** | 单次请求端到端 |

## 二、两个项目对比

### 架构层面

| 维度 | Go 项目 (ChatGpt-Image-Studio) | Python 项目 (chatgpt2api) |
|------|------|------|
| 并发模型 | goroutine (2-4KB/个) | threading (8MB/个) |

| 单任务多图 | 最多2个unit并行 | 串行 |
| 账号路由 | 分组/优先级/reserve百分比 | 简单轮询 |

### 出图速度

| 维度 | Go 项目 | Python 项目 |
|------|---------|------------|

| Poll 间隔 | 3秒 | 10秒 |
| 初始等待 | 无 | 10秒 |
| HTTP 客户端 | Go net/http + uTLS | curl_cffi (重) |

| 维度 | Go 项目 | Python 项目 |
|------|---------|------------|

## 三、是否需要 Redis

### 单实例部署（<1万并发）
**不需要 Redis**。Go 项目本地 channel + mutex 足够。

### 多实例部署（10万并发）
**必须 Redis**，原因：

1. **账号锁**：多个实例同时选账号，不用 Redis 会导致同一账号被多个实例同时使用
2. **任务去重**：同一个 task_id 不能在多个实例上重复执行
3. **配置同步**：config_backend=redis 让所有实例共享配置
4. **状态共享**：账号 quota、限流状态需要跨实例可见

### 10万并发的部署架构

```
                    ┌─────────────┐
                    │  Nginx/LB   │  ← 负载均衡
                    └──────┬──────┘
           ┌───────────────┼───────────────┐
           ▼               ▼               ▼
    ┌─────────────┐ ┌─────────────┐ ┌─────────────┐
    │  Go 实例 1  │ │  Go 实例 2  │ │  Go 实例 N  │
    │  (1万并发)  │ │  (1万并发)  │ │  (1万并发)  │
    └──────┬──────┘ └──────┬──────┘ └──────┬──────┘
           │               │               │
           └───────────────┼───────────────┘
                           ▼
                    ┌─────────────┐
                    │    Redis    │  ← 账号锁 + 配置 + 状态
                    └─────────────┘
```

每个 Go 实例可以轻松撑 1万并发 goroutine。10个实例 = 10万并发。

## 四、需要二改的点

### 必改（核心瓶颈）

| # | 改动 | 原因 | 难度 |
|---|------|------|------|
| 1 | **Redis 分布式账号锁** | 多实例部署必须，防止同一账号被多实例同时使用 | 中 |
| 2 | **max_image_concurrency 提升** | 默认8，需要改到 1000-10000 | 低 |
| 3 | **image_queue_limit 提升** | 默认32，需要改到 100000 | 低 |
| 4 | **maxImageTaskParallelUnitsPerTask 提升** | 默认2，改到 4-8，单任务多图并行 | 低 |
| 5 | **账号预热池** | bootstrap + requirements 提前做好缓存，请求来了直接用 | 高 |

### 建议改（提升体验）

| # | 改动 | 原因 | 难度 |
|---|------|------|------|
| 6 | 账号智能路由 | 按成功率/响应时间加权选择，而非简单轮询 | 中 |
| 7 | 请求优先级队列 | VIP 请求插队 | 中 |
| 8 | 详细诊断日志 | 从 Python 项目搬 stage_timings 体系 | 低 |
| 9 | Poll 智能退避 | 从 Python 项目搬 Retry-After 感知 | 低 |
| 10 | WebSocket 替代前端轮询 | 减少连接数，实时推送任务状态 | 中 |

### 已发现的 Bug：账号类型识别问题

**现象**：Plus 账号出图报 `chatgpt account id is required`

**根因**：Go 项目根据账号 `type` 字段选择出图路径：
```go
// server.go:2248
func (s *Server) configuredImageRoute(accountType string) string {
    switch accountType {
    case "Plus", "Pro", "Team":
        return paid_image_route  // 默认 "responses"，需要 account_id
    default:
        return free_image_route  // 默认 "legacy"，不需要 account_id
    }
}
```

**问题**：
1. `responses` 路径需要 `account_id`，但很多导入的账号 `account_id` 为空
2. 如果账号类型识别错误（比如 Free 被识别为 Plus），会走错路径
3. CPA 导入的账号 type 字段可能不规范

**临时解决**：`config.toml` 中设置 `paid_image_route = "legacy"`
**根本解决**：二改时统一走 legacy 路径，或在选路径前校验 account_id 是否存在

## 五、10万并发的数学

### 账号需求计算

- 每次生图耗时：~60秒（平均）
- 每账号并发：1（ChatGPT 限制）
- 10万并发 = 需要 **10万个 free 账号同时在线**

### Free 账号额度

- Free 账号每天图片配额：约 2-3 张（不确定，可能更少）
- 10万并发持续运行 1 小时 = 10万张图
- 需要的账号数：10万 / 2 = **5万个账号**（如果每个账号只能出2张）

### 服务器资源

- 每个 Go 实例：1万并发 ≈ 2GB 内存 + 4核 CPU
- 10个实例：20GB 内存 + 40核
- Redis：8GB 内存（存10万账号状态）
- 带宽：每张图 ~2MB，10万张/小时 = ~55GB/小时 ≈ 125Mbps
